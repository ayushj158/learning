# 🧠 Service Blueprint vs Journey Map

### **Long-term Revision Notes (Sticky Memory Version)**

---

## 🧩 1. One-Line Definitions (memorize these)

### **Journey Map**

> A **Journey Map** shows **how a customer experiences a service over time**, focusing on **touchpoints, thoughts, emotions, and pain points**.

### **Service Blueprint**

> A **Service Blueprint** shows **how the service is actually delivered**, mapping **customer actions + frontstage interactions + backstage processes + systems**.

📌 **Shortcut memory hook:**

* Journey Map = **Experience**
* Service Blueprint = **Execution**

---

## 🎯 2. Primary Purpose (WHY they exist)

| Artifact              | Core Purpose                                                   |
| --------------------- | -------------------------------------------------------------- |
| **Journey Map**       | Discover experience gaps & improvement opportunities           |
| **Service Blueprint** | Make the experience **operationally and technically feasible** |

👉 Journey Map finds **what’s broken**
👉 Service Blueprint explains **what must change to fix it**

---

## 🧠 3. Mental Model (How to visualize them)

### Journey Map = **Timeline**

```
Discover → Explore → Decide → Use → Support
```

### Service Blueprint = **Layered Stack**

```
Customer Actions
───────────────  (Line of Interaction)
Frontstage (UI / Agents / App)
───────────────  (Line of Visibility)
Backstage Processes
───────────────
Systems / APIs / Teams
```

📌 If you can draw **lines** → you’re thinking Blueprint
📌 If you can draw **phases** → you’re thinking Journey

---

## 🔍 4. What Each Artifact Contains (Checklist)

### 🧭 Journey Map Includes

* Stages / phases of the journey
* User goals at each stage
* Touchpoints (app, branch, call, email, etc.)
* Emotions (happy, frustrated, anxious)
* Pain points & opportunities

❌ Does NOT include:

* APIs
* Internal workflows
* System dependencies

---

### 🧩 Service Blueprint Includes

* Customer actions
* Frontstage interactions (UI, agents, chatbots)
* Backstage actions (manual + automated)
* Systems, services, integrations
* Ownership (teams, roles)
* Dependencies & failure points

❌ Does NOT focus on:

* Emotional depth
* Narrative storytelling

---

## 🧪 5. Simple Example (Ultra-condensed)

### Example: **Bank Account Opening**

#### Journey Map view

> “Customer applies online, feels confused during KYC, relieved after approval.”

#### Service Blueprint view

> “KYC triggers document service → AML check → manual review → core banking API → notification service.”

📌 Same service.
📌 Two very different lenses.

---

## 🧑‍🤝‍🧑 6. Who Uses What (in real orgs)

| Role            | Uses Journey Map | Uses Service Blueprint |
| --------------- | ---------------- | ---------------------- |
| UX / Design     | ✅ Heavy          | ✅ Medium               |
| Product Manager | ✅                | ✅                      |
| Engineering     | ❌ Rare           | ✅ Heavy                |
| Architects      | ❌                | ✅                      |
| Operations      | ❌                | ✅                      |

📌 **If developers care about it → it’s probably a Blueprint**

---

## 🏗️ 7. When to Create Each (Sequence)

### Correct Order (important!)

1️⃣ **Journey Map** → Understand experience
2️⃣ **Service Blueprint** → Design delivery
3️⃣ Architecture / APIs / Backlogs → Build

❌ Blueprint without Journey = efficient delivery of a bad experience
❌ Journey without Blueprint = great idea that never ships

---

## 🔁 9. One-Sentence Comparison (Ultimate Recall)

> **Journey Maps describe the customer’s story; Service Blueprints describe the organization’s response to that story.**

If you remember **only this line**, you’ll be fine years later.

---

## 🧠 10. Ultra-Short Revision (30-second skim)

* Journey = **Customer lens, emotional, phased**
* Blueprint = **Operational lens, layered, technical(customer actions -> frontstage interactions -> backend processes -> systems)**
* Journey finds **what to fix**
* Blueprint explains **how to fix it**
* Journey first, Blueprint second
* Both together = buildable experience

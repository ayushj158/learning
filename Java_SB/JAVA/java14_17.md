# 🟦 JAVA 14–17 — RECORDS, SEALED CLASSES, PATTERN MATCHING (DEEP DIVE)

---

# 1️⃣ RECORDS (Java 16)

---

## 1. Why Records Were Introduced (Real Problem)

Before records, Java data carriers looked like this:

```java
class Money {
    private final BigDecimal amount;
    private final Currency currency;

    // constructor
    // getters
    // equals
    // hashCode
    // toString
}
```

Problems:

* Boilerplate dominates intent
* Easy to make mistakes
* Hard to spot *what the class really represents*

Java needed:

> **A first-class way to model immutable data**

---

## 2. What a Record REALLY Is (Mental Model)

> **A record is a transparent, immutable data carrier.**

Key properties:

* Immutable by default
* Value-based equality
* State is the API
* No identity beyond its data

---

## 3. Basic Record Example

```java
public record Money(BigDecimal amount, Currency currency) {}
```

The compiler generates:

* private final fields
* constructor
* accessors
* equals / hashCode
* toString

Correct by default.

---

## 4. Canonical Constructor (VERY IMPORTANT)

You can enforce invariants:

```java
public record Money(BigDecimal amount, Currency currency) {
    public Money {
        if (amount == null || currency == null) {
            throw new IllegalArgumentException();
        }
        if (amount.signum() < 0) {
            throw new IllegalArgumentException("Negative money");
        }
    }
}
```

Senior insight:

> Records are immutable, **not invariant-free**.

---

## 5. What Records Are PERFECT For (FS GOLD)

Use records for:

* Value Objects (Money, AccountId)
* DTOs
* API payloads
* Events
* Query results
* Snapshots

In banks:

* Ledger entries
* Audit events
* Pricing results

---

## 6. What Records Are BAD At (CRITICAL)

Do NOT use records for:

* JPA entities
* Objects with lifecycle
* Objects with identity
* Objects with behaviour-heavy logic

❌ Bad:

```java
record Account(...) { // ❌ }
```

Why?

* Accounts evolve
* Have identity
* Have invariants across time

---

## 7. Records vs Lombok (Interview Insight)

| Aspect               | Record | Lombok   |
| -------------------- | ------ | -------- |
| Language feature     | Yes    | No       |
| Immutable by default | Yes    | Optional |
| Clear intent         | Strong | Mixed    |
| Tooling              | Native | Plugin   |

Senior stance:

> Prefer records where semantics fit; Lombok where flexibility is needed.

# 2️⃣ SEALED CLASSES (Java 17)

---

## 8. Why Sealed Classes Were Introduced

Before sealed classes:

```java
interface PaymentMethod {}
```

Anyone can implement it.
Anywhere.
Anytime.

Problems:

* Domain explosion
* Invalid implementations
* Hard-to-reason exhaustiveness

Java needed:

> **Controlled inheritance**

---

## 9. What Sealed Classes REALLY Are

> **A sealed class explicitly controls who can extend it.**

```java
public sealed interface PaymentMethod
    permits CardPayment, BankTransfer, Cash {}
```

Only these types are allowed.

---

## 🔟 Why This Matters for Domain Modelling

In financial systems:

* Domains are closed sets
* Regulations define allowed states
* Unknown subtypes = bugs

Sealed classes enforce:

* Compile-time exhaustiveness
* Safer refactoring
* Clear domain boundaries

---

## 1️⃣1️⃣ Sealed Classes + Business Rules

```java
sealed interface AccountStatus
    permits Active, Frozen, Closed {}
```

This prevents:

* “TemporaryActive”
* “LegacyClosedButNotReally”
* Other nonsense states

---

## 1️⃣2️⃣ Sealed vs Enum (Important Nuance)

| Feature       | Enum     | Sealed       |
| ------------- | -------- | ------------ |
| State         | Constant | Rich objects |
| Behaviour     | Limited  | Full         |
| Extensibility | Closed   | Controlled   |
| Polymorphism  | Weak     | Strong       |

Rule:

> Use enums for **simple states**, sealed types for **rich behaviour**.

---

## 1️⃣3️⃣ Where NOT to Use Sealed Classes

Avoid sealed classes when:

* You expect external extensions
* You build frameworks
* You expose public APIs to unknown consumers

Sealed = **closed world assumption**.

# 3️⃣ PATTERN MATCHING (Java 16–17)

---

## 1️⃣4️⃣ Why Pattern Matching Exists

Before:

```java
if (obj instanceof Account) {
    Account a = (Account) obj;
}
```

Problems:

* Verbose
* Unsafe
* Easy to forget cast

Java wanted:

> **Safer, more expressive type inspection**

---

## 1️⃣5️⃣ Pattern Matching for `instanceof`

```java
if (obj instanceof Account a) {
    process(a);
}
```

Benefits:

* No cast
* Scoped variable
* Safer refactoring

---

## 1️⃣6️⃣ Pattern Matching + Sealed Classes (POWER COMBO)

```java
return switch (paymentMethod) {
    case CardPayment c -> processCard(c);
    case BankTransfer b -> processTransfer(b);
    case Cash cash -> processCash(cash);
};
```

Key:

* Compiler enforces exhaustiveness
* No default required
* If you add a subtype → compiler error

This is **huge** for correctness.

---

## 1️⃣7️⃣ Why Banks LOVE This Combination

This combo:

* Prevents missing cases
* Makes rules explicit
* Eliminates runtime surprises
* Improves auditability

In regulated domains:

> **Compile-time safety beats runtime checks.**

---

# 4️⃣ FS / ENTERPRISE REALITY CHECK

---

## 1️⃣8️⃣ Adoption Reality in Banks

Typical pattern:

* Records → adopted quickly
* Pattern matching → adopted gradually
* Sealed classes → used selectively in core domain

Why selective?

* Framework compatibility
* Serialization concerns
* JPA limitations

---

## 1️⃣9️⃣ Migration Strategy (Practical)

1. Start with records for DTOs/events
2. Introduce pattern matching where switch-heavy code exists
3. Use sealed classes in **new domain models only**

Never retrofit sealed classes aggressively into legacy models.
